# Planning

## 适用场景

多步骤编码工作需要清晰的执行计划：阶段拆解、验收条件、进度跟踪。适合在动手实现前规划。

## 核心原则

1. **计划先于行动**：复杂任务先规划再执行，避免边做边改方向。
2. **阶段可验证**：每个阶段结束时必须有明确的验证点。
3. **依赖有序**：阶段按依赖关系排序，前置完成才能开始后置。
4. **留缓冲**：计划要为未知和返工预留空间。
5. **可调整**：计划是活的，发现新信息要更新。

## 执行流程

### 1. 明确目标与范围
- 终点定义：完成时系统是什么状态
- 验收标准：如何判断完成（配合 `acceptance-criteria`）
- 范围边界：明确不做什么

### 2. 拆解为阶段
- **按依赖拆**：识别哪些工作必须先做
- **按价值拆**：先做最高价值的部分
- **按风险拆**：高风险的先验证
- 每阶段产出一个可验证的里程碑

### 3. 定义每阶段
对每个阶段明确：
- **输入**：开始时需要什么
- **任务**：这个阶段要完成的具体工作
- **输出**：产出的代码/文档/测试
- **验证**：如何确认这个阶段完成

### 4. 排序与依赖
画依赖图，确认执行顺序。并行可做的标注出来。

### 5. 估时与缓冲
- 每阶段粗略估时
- 留 20-30% 缓冲给未知问题
- 标注高风险阶段

### 6. 跟踪与调整
执行中：
- 完成的阶段标记 ✅
- 偏离计划时更新计划（而非硬撑原计划）
- 记录阻塞和决策

## 计划文档结构

```markdown
# 执行计划：<任务名>

## 目标
## 验收标准
## 范围（做 / 不做）

## 阶段
### 阶段 1：<名>
- 任务：
- 输出：
- 验证：
- 依赖：无

### 阶段 2：<名>
- 依赖：阶段 1
...

## 风险
## 时间估算
```

## 检查清单

- [ ] 终点和验收标准明确
- [ ] 拆解为可验证的阶段
- [ ] 阶段依赖关系清晰
- [ ] 每阶段有明确的验证方式
- [ ] 高风险阶段已标注
- [ ] 留有缓冲

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 没计划直接编码 | 先规划再执行 |
| 阶段无法验证 | 每阶段有验证点 |
| 忽略依赖 | 按依赖排序 |
| 计划过细（到小时级） | 阶段级粒度即可 |
| 死守原计划 | 发现偏差及时调整 |

## 输出要求

1. **目标与验收标准**
2. **阶段清单**：每阶段的任务/输出/验证/依赖
3. **依赖图**：阶段间的关系
4. **风险标注**
5. **时间估算**
