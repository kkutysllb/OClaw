# Architecture Design

## 适用场景

进行模块设计、边界划分、依赖方向规划、服务拆分、核心引擎设计、插件系统设计、横切关注点决策等架构级技术决策。

## 核心原则

1. **边界先于实现**：先明确模块边界和职责，再写代码。边界模糊必然导致耦合。
2. **依赖方向单向**：依赖应指向稳定方（接口/核心），不指向易变方（实现/细节）。
3. **高内聚低耦合**：同模块内的代码强相关，跨模块依赖最小化。
4. **为变更而设计**：预判最可能的变更维度，让架构在该方向上易于扩展。
5. **简单性优先**：能不用抽象就不用。过度设计比设计不足更危险。

## 执行流程

### 1. 理解约束与需求
- 梳理功能需求：系统要做什么
- 梳理非功能需求：性能、可用性、可扩展性、安全
- 梳理约束：团队能力、技术栈、现有系统、时间预算
- `read_file_lines` / `search_code` 了解现有代码结构和约定

### 2. 识别核心域与边界
- **领域驱动**：按业务领域划分模块，而非按技术层
- **限界上下文**：每个模块有明确的职责边界和领域语言
- **识别聚合**：哪些实体必须保持一致性，归为同一聚合
- 输出：模块清单 + 每个模块的一句话职责

### 3. 定义接口与依赖
- **接口隔离**：模块通过明确接口通信，不暴露内部实现
- **依赖方向**：画依赖图，确保无环、指向抽象
- **通信方式**：同步调用 / 事件 / 消息队列，按耦合度需求选择
- 输出：接口契约 + 依赖图

### 4. 设计横切关注点
统一规划跨模块的关注点：
- **错误处理**：统一异常层级和错误传播策略
- **日志与可观测性**：统一的日志格式、metrics、tracing
- **配置管理**：环境变量、配置加载、热更新策略
- **安全**：认证、授权、加密的统一方案

### 5. 评估与权衡
对每个关键决策记录权衡：
| 决策点 | 选项 A | 选项 B | 选择 | 理由 |
|---|---|---|---|---|

### 6. 文档化
输出架构文档：
- **C4 Context/Container 图**：系统级上下文
- **模块依赖图**：内部结构
- **ADR（架构决策记录）**：关键决策及理由

## 常见架构模式选择

| 场景 | 推荐模式 | 注意事项 |
|---|---|---|
| 单体应用，团队小 | 分层架构（router/service/persistence） | 保持层间单向依赖 |
| 多团队协作 | 模块化单体 → 微服务 | 先模块化再拆服务 |
| 插件化扩展 | 插件接口 + 注册机制 | 接口稳定，插件可独立加载 |
| 事件驱动 | 事件总线 + 消费者 | 注意最终一致性和幂等 |
| 读多写少 | CQRS（读写分离） | 读模型可独立优化 |

## 架构坏味道

| 信号 | 问题 |
|---|---|
| 循环依赖 | 模块边界划分错误 |
| 改一处动多处 | 职责分散（霰弹修改） |
| 上帝类/上帝模块 | 职责未拆分 |
| 跨层直接调用 | 绕过了抽象边界 |
| 到处是 adapter/wrapper | 过度设计 |
| 一个接口被 20 个类实现 | 接口过大，应拆分 |

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 了解现有结构 | `search_code` / `find_files` | 摸清代码组织 |
| 读核心模块 | `read_file_lines` | 理解现有设计 |
| 查依赖关系 | `search_code "<import>"` | 依赖图分析 |
| 记录决策 | `write_file` | 写 ADR 文档 |

## 检查清单

- [ ] 每个模块有明确的一句话职责
- [ ] 依赖方向单向，无循环
- [ ] 接口契约明确，实现可替换
- [ ] 横切关注点有统一方案
- [ ] 关键决策有权衡记录（ADR）
- [ ] 架构文档已产出（图 + 说明）
- [ ] 没有过度设计（能简单则简单）

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 边界模糊的"大模块" | 明确职责边界的聚焦模块 |
| 双向依赖 | 单向依赖，接口隔离 |
| 为所有可能扩展设计 | 为最可能的变更设计 |
| 跳过文档直接编码 | 先出架构图和 ADR |
| 照搬流行架构 | 基于实际约束选择 |

## 输出要求

1. **架构概览**：模块清单 + 一句话职责
2. **依赖图**：模块间依赖方向（文字描述或 mermaid）
3. **接口契约**：核心模块的公共接口
4. **横切方案**：错误/日志/配置/安全的统一策略
5. **关键 ADR**：重要决策的选项、权衡、选择理由
6. **迁移路径**（如适用）：从现状到目标架构的渐进步骤
