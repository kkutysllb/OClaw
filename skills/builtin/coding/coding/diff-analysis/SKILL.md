# Diff Analysis

## 适用场景

分析代码 diff：理解变更范围、评估文件级风险、对比前后代码、为 review 提供基于具体改动的 grounded 发现。强调"基于实际改动行"而非泛泛而谈。

## 核心原则

1. **基于具体行**：所有发现必须指向具体的 changed hunk，不空谈。
2. **区分变更类型**：新增/修改/删除/移动，每种的风险评估不同。
3. **文件级风险分层**：核心模块的小改 > 边缘模块的大改。
4. **关注净变更**：格式化导致的大 diff 但净逻辑不变 ≠ 高风险。
5. **追溯意图**：从 commit message 和 diff 推断变更目的，判断是否达成。

## 执行流程

### 1. 获取 Diff
- `git_diff`：工作区未提交改动
- `git_show <commit>`：特定提交的改动
- `git_diff <base>..<head>`：分支间对比（PR review 用）
- 指定 `--stat` 先看文件级概览，再逐文件看 hunk

### 2. 文件级分类
按变更性质分组：
| 类型 | 特征 | 风险 |
|---|---|---|
| **新增文件** | 全新逻辑 | 中（需全面 review） |
| **核心修改** | 改动已有业务逻辑 | 高（可能影响现有行为） |
| **配置变更** | 改 config/env/CI | 高（影响环境） |
| **测试变更** | 改测试代码 | 低（除非删测试） |
| **文档/注释** | 只改文档 | 极低 |
| **格式化** | 大量行变但逻辑不变 | 低（注意噪声） |
| **删除文件** | 移除代码 | 中（确认是否误删） |

### 3. Hunk 级分析
对每个变更 hunk：
- **读完整上下文**：`read_file_lines` 读取 hunk 周围 20+ 行
- **识别变更性质**：新增逻辑 / 修改条件 / 删除分支 / 重命名
- **评估影响面**：`search_code` 查被改符号的调用方
- **检查边界**：新增分支是否覆盖所有条件？删除的逻辑是否有替代？

### 4. 风险评估
对每个高风险 hunk 追问：
- [ ] 这个改动会破坏现有调用方吗？
- [ ] 新增逻辑的所有分支都处理了吗？
- [ ] 删除的代码有其他地方依赖吗？
- [ ] 配置变更是否影响其他环境？
- [ ] 是否有对应的测试覆盖这次变更？

### 5. 输出发现
按风险分级，每条发现必须指向具体文件和行：
```
[高风险] src/auth/login.py:45-50
  变更：移除了 password 为空时的校验分支
  风险：调用方可能传入空密码导致绕过认证
  建议：恢复校验或确认调用方已保证非空
```

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 工作区改动 | `git_diff` | 默认 diff |
| 特定提交 | `git_show` | 查单个 commit |
| 分支对比 | `git_diff base..head` | PR review |
| 读上下文 | `read_file_lines` | hunk 周围代码 |
| 查调用方 | `search_code` | 影响面分析 |
| 提交历史 | `git_log` | 变更意图 |

## 检查清单

- [ ] 已获取完整 diff（含 stat 概览 + 逐 hunk）
- [ ] 文件按变更类型和风险分层
- [ ] 每个高风险 hunk 都读取了完整上下文
- [ ] 被改符号的调用方已搜索评估
- [ ] 每条发现都指向具体文件和行号
- [ ] 变更意图（从 commit message）已确认
- [ ] 测试覆盖情况已检查

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 泛泛评论"代码质量" | 指向具体 changed line |
| 忽略删除的代码 | 检查删除逻辑的依赖 |
| 把格式化当大改 | 区分净逻辑变更 |
| 只看新增不看修改 | 修改已有逻辑风险更高 |
| 不读上下文只看 hunk | 读周围 20+ 行理解全貌 |
| 忽略配置/CI 变更 | 配置变更往往高风险 |

## 输出要求

1. **Diff 概览**：文件数、增删行数、变更类型分布
2. **文件级风险表**：每个文件的类型 + 风险等级 + 一句话说明
3. **高风险 hunk 详析**：具体行号 + 变更内容 + 风险 + 建议
4. **影响面评估**：被改符号的调用方清单
5. **测试覆盖检查**：哪些变更有测试、哪些没有
6. **总体结论**：这次变更是否安全、是否达到 commit message 声明的目的
