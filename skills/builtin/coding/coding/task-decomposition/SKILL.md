# Task Decomposition

## 适用场景

将宽泛的编码请求拆解为可执行的子任务：任务 ID 划分、阶段化工作、实现分期、独立子任务。

## 核心原则

1. **单一职责**：每个子任务只做一件事，可独立描述和验证。
2. **可独立验证**：子任务完成后必须有明确的验证方式。
3. **依赖最小化**：尽量让子任务独立，减少互相阻塞。
4. **粒度适中**：太粗无法管理，太细增加协调成本。
5. **可并行标注**：独立的子任务标记为可并行，提升效率。

## 执行流程

### 1. 理解整体目标
- 明确最终交付物是什么
- 确认验收标准
- 理解约束（时间、技术、依赖）

### 2. 识别工作维度
将大任务按维度切分：
- **按层次**：数据层 → 服务层 → 接口层 → UI 层
- **按功能**：功能 A / 功能 B / 功能 C
- **按阶段**：基础架构 → 核心功能 → 增强 → 优化
- **按类型**：实现 / 测试 / 文档 / 配置

### 3. 拆解为子任务
每个子任务定义：
- **ID**：唯一标识（如 task-001）
- **标题**：一句话描述
- **描述**：具体要做什么
- **输入**：需要什么前置
- **输出**：产出什么
- **验证**：如何确认完成
- **依赖**：依赖哪些其他子任务
- **可并行**：是否可与某些子任务并行

### 4. 排序与分组
- 按依赖关系排序
- 标记可并行的子任务组
- 识别关键路径（最长依赖链）

### 5. 输出任务清单
```markdown
## 子任务清单

### task-001：数据模型定义
- 描述：定义核心实体和关系
- 输出：models.py + migration
- 验证：migration 可执行，模型可实例化
- 依赖：无
- 可并行：否（是其他任务的前置）

### task-002：用户注册接口
- 描述：实现 POST /users/register
- 输出：router + service + test
- 验证：测试通过，返回 201
- 依赖：task-001
- 可并行：与 task-003 并行

### task-003：用户登录接口
- 描述：实现 POST /users/login
- 依赖：task-001
- 可并行：与 task-002 并行
```

## 拆解粒度参考

| 任务规模 | 建议拆解 |
|---|---|
| 单文件小改 | 不用拆，直接做 |
| 单模块多文件 | 拆 2-4 个子任务 |
| 跨模块功能 | 拆 5-10 个子任务 |
| 项目级 | 按阶段拆，每阶段再拆子任务 |

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 评估代码现状 | `read_file_lines` / `search_code` | 拆解基于现状 |
| 澄清范围 | `ask_clarification` | 确认边界 |
| 记录任务 | `write_file` | 产出任务清单 |

## 检查清单

- [ ] 每个子任务单一职责
- [ ] 每个子任务可独立验证
- [ ] 依赖关系明确且最小化
- [ ] 可并行的子任务已标注
- [ ] 粒度适中（不太粗不太细）
- [ ] 关键路径已识别

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 子任务依赖一团乱 | 依赖最小化，DAG 清晰 |
| 子任务无法验证 | 每个有明确验证方式 |
| 粒度太细（到函数级） | 聚焦可独立交付的单元 |
| 忽略并行机会 | 标注可并行子任务 |
| 拆解后不跟踪 | 用清单跟踪完成状态 |

## 输出要求

1. **子任务清单**：ID + 标题 + 描述 + 输出 + 验证 + 依赖
2. **依赖图**：子任务间关系
3. **并行分组**：哪些可同时进行
4. **关键路径**：最长依赖链
5. **执行顺序建议**
