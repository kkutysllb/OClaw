# QA Test Plan

## 适用场景

- 功能开发完成，需要制定系统化的测试计划
- 版本发布前需要创建验证清单和回归测试矩阵
- 需要定义浏览器兼容性测试范围
- 需要协调手动测试和自动化测试的覆盖分工

## 核心原则

1. **覆盖优先级**：按风险等级排序测试项——核心路径 > 高频路径 > 边界场景
2. **可追溯**：每个需求都有对应的测试用例，确保需求覆盖率
3. **分层测试**：单元 → 集成 → E2E，不同层级覆盖不同风险
4. **现实约束**：在有限时间内最大化测试价值，优先测最可能出错的
5. **明确验收标准**：每个测试用例有明确的预期结果和通过条件

## 执行流程

### 1. 分析需求

- 列出本次变更涉及的所有功能点和受影响的模块
- 识别高风险区域（安全、支付、数据迁移、并发）
- 确定测试范围边界（测什么、不测什么）

### 2. 构建测试矩阵

| 功能点 | 单元测试 | 集成测试 | E2E | 手动验证 | 风险等级 |
|-------|---------|---------|-----|---------|---------|
| 用户登录 | ✅ | ✅ | ✅ | ✅ | High |
| 数据导出 | ✅ | ✅ | — | ✅ | Medium |
| UI 主题切换 | — | — | ✅ | ✅ | Low |
| API 分页 | ✅ | ✅ | — | — | Medium |

### 3. 浏览器兼容性矩阵

| 浏览器 | 版本 | 桌面 | 移动 | 优先级 |
|-------|------|------|------|-------|
| Chrome | 最新 | ✅ | ✅ | P0 |
| Safari | 最新 | ✅ | ✅ | P0 |
| Firefox | 最新 | ✅ | — | P1 |
| Edge | 最新 | ✅ | — | P2 |

### 4. 编写测试用例

每个用例包含：

```
用例 ID: TC-001
标题: 用户使用有效凭据登录
前置条件: 用户已注册，dev server 运行中
步骤:
  1. 打开登录页面
  2. 输入有效邮箱和密码
  3. 点击登录按钮
预期结果:
  - 跳转到首页
  - 显示用户名
  - 控制台无错误
优先级: P0
类型: 正向
```

### 5. 测试用例分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **正向测试** | 验证正常流程 | 有效输入 → 正确输出 |
| **反向测试** | 验证异常处理 | 无效输入 → 友好错误提示 |
| **边界测试** | 验证边界值 | 空字符串/最大长度/零值 |
| **回归测试** | 验证未改动的旧功能 | 修改登录后验证注册仍正常 |
| **安全测试** | 验证安全防护 | 越权访问/SQL 注入/XSS |
| **性能测试** | 验证响应时间 | 大数据量加载/并发请求 |

### 6. 发布验证清单

- [ ] 所有 P0 测试用例通过
- [ ] 所有 P1 测试用例通过（或已知问题可接受）
- [ ] 无 Critical/High 级别未修复 bug
- [ ] 自动化测试套件全绿
- [ ] 性能指标达标（首屏 < 3s，API < 500ms）
- [ ] 安全扫描通过
- [ ] 回归测试通过（核心功能无退化）

### 7. 回归测试策略

- 维护核心路径自动化测试（每次提交运行）
- 发布前执行完整回归测试套件
- 记录已知问题，评估是否可接受上线

## 工具优先级

| 工具 | 用途 |
|------|------|
| `write_file` | 编写测试计划文档 |
| `run_tests` | 执行自动化测试套件 |
| Bash（Playwright） | 执行 E2E 测试 |
| `git_diff` / `git_status` | 识别变更范围，确定回归测试重点 |

## 检查清单

- [ ] 列出了所有受影响的功能点
- [ ] 每个功能点有对应的测试用例
- [ ] 测试矩阵覆盖单元/集成/E2E/手动
- [ ] 定义了浏览器兼容性范围
- [ ] 用例包含正向、反向、边界、安全分类
- [ ] 定义了发布验收标准（P0 全通过等）
- [ ] 回归测试策略已明确

## 反模式

| ❌ 避免 | ✅ 应该 |
|---------|--------|
| "测一下"没有计划 | 系统化测试矩阵 + 用例清单 |
| 只测正向流程 | 覆盖反向、边界、异常场景 |
| 不分优先级全测 | 按风险排序，优先 P0 |
| 不记录已知问题 | 记录并评估是否可接受上线 |
| 发布前才写测试计划 | 开发阶段就准备测试用例 |

## 输出要求

1. 提供完整的测试矩阵（功能 × 测试类型 × 优先级）
2. 提供浏览器兼容性矩阵
3. 列出关键测试用例（至少覆盖 P0/P1）
4. 提供发布验证清单
5. 标注自动化 vs 手动的分工
