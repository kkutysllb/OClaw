# Systematic Debugging

## 适用场景

需要系统化的根因分析来定位 bug：崩溃、测试失败、非预期行为、性能瓶颈、集成故障、前后端错误。强调"先定位再修复"的纪律性流程，避免盲目试错。

## 核心原则

1. **复现优先**：无法复现就无法确认修复。第一步永远是建立稳定复现路径。
2. **假设-验证循环**：提出明确假设 → 设计验证 → 观察结果 → 证实/证伪 → 迭代。
3. **一次只改一个变量**：同时改多处会让"哪个修复生效"变成谜题。
4. **二分法是利器**：大范围不确定时，注释/禁用一半快速缩小可疑区。
5. **读代码优于猜代码**：用工具读取实际实现，不要凭记忆推断行为。
6. **3 次失败就停下**：同一方法连续失败 3 次，停止盲试，重新分析。

## 执行流程

### 1. 稳定复现（Reproduce）
- 构造最小复现：最少输入、最短路径
- 记录环境：OS、语言版本、依赖版本、配置
- 优先写成失败测试（既是复现也是回归保护）
- 如果无法 100% 复现，记录触发概率和条件

### 2. 采集证据（Observe）
- **错误信息全文**：不要截断，完整记录
- **堆栈跟踪**：从顶层到底层逐帧理解
- **日志**：bug 发生前后的完整日志
- **状态快照**：关键变量的值、数据流中间状态

### 3. 形成假设（Hypothesize）
基于证据，写出明确的因果假设：
> "当 `user_id` 为 null 时，第 45 行的 `user.name` 访问抛出 AttributeError，
> 因为 `get_user()` 在缓存未命中时返回了 None 而非抛出 NotFound。"

假设必须：
- 可证伪（能设计一个验证来否定它）
- 具体到代码位置和条件
- 解释所有已观察到的症状（不能只解释一部分）

### 4. 验证假设（Test）
选择最低成本的验证方式：
- **加日志/断言**：在可疑点打印中间状态，运行复现
- **写探针测试**：直接调用可疑函数，验证输入输出
- **临时修改**：强行修正假设的条件，看症状是否消失
- **二分注释**：禁用一半代码，看 bug 是否消失

### 5. 定位根因（Conclude）
- 假设被验证后，追溯根因到最源头（不是中间表现）
- 根因通常是：错误的假设、缺失的校验、边界未处理、状态污染
- 确认根因能解释：为什么会发生、为什么在此处表现、为什么之前没发现

### 6. 最小修复（Fix）
- 只修复根因，不顺手改其他
- 删除所有临时调试代码（日志/断言/print）
- 用 `apply_diff` 做精准编辑

### 7. 回归验证（Regression）
- 复现测试现在通过
- `run_tests` 全套通过，无新增失败
- 思考：同类问题是否可能在别处也存在？是否需要防御性加固？

## 调试决策树

```
bug 发生
  │
  ├─ 能复现吗？
  │    ├─ 是 → 进入定位
  │    └─ 否 → 加日志/扩大可观测性 → 等待再次发生
  │
  ├─ 有堆栈/错误信息吗？
  │    ├─ 是 → 直奔出错点读取代码
  │    └─ 否 → 是静默错误？加断言/日志暴露
  │
  ├─ 是回归吗（之前好的）？
  │    ├─ 是 → git log + bisect 找引入 commit
  │    └─ 否 → 全新 bug，走假设验证流程
  │
  └─ 影响范围多大？
       ├─ 单点 → 针对性修复
       └─ 多处 → 根因在公共依赖/基础设施
```

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 读出错代码 | `read_file_lines` | 精准读取堆栈指向位置 |
| 搜索符号/错误 | `search_code` | 找定义点和抛出点 |
| 查最近改动 | `git_log` / `git_diff` | 排查回归 |
| 运行复现 | `bash` | 执行命令/脚本 |
| 加日志后运行 | `bash` | 观察运行时状态 |
| 验证修复 | `run_tests` | 确认修复且无回归 |

## 检查清单

- [ ] 已建立稳定（或高概率）复现路径
- [ ] 完整记录了错误信息和堆栈
- [ ] 形成了可证伪的明确假设
- [ ] 假设被验证（日志/断言/探针证据）
- [ ] 根因追溯到最源头，非中间表现
- [ ] 修复是最小改动，未夹带无关变更
- [ ] 临时调试代码已清理
- [ ] 复现测试通过 + 全套测试无回归

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 不复现就改代码猜测 | 先建立复现路径 |
| 同时改多处碰运气 | 一次只改一个变量 |
| 只看症状不追根因 | 追溯到最源头 |
| 改完不删调试代码 | 清理所有 print/日志 |
| 连续试错不改策略 | 3 次失败就停下重新分析 |
| 修完不验证回归 | run_tests 确认无副作用 |

## 输出要求

1. **复现路径**：最小复现步骤或测试
2. **根因**：一句话因果说明（当 X 发生时，Y 导致 Z）
3. **验证证据**：证明假设成立的日志/断言输出
4. **修复**：改了什么、为什么这样改
5. **回归结果**：复现测试通过 + 全套测试通过的输出
6. **防御建议**：如何避免同类问题复发
