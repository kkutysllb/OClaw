# Acceptance Criteria

## 适用场景

定义完成条件、可测试的验收标准、QA 检查项、发布门禁。确保"做完"有明确且可验证的定义。

## 核心原则

1. **可测试**：每条验收条件必须能转化为一个明确的通过/失败判断。
2. **无歧义**：用"当 X 时，系统应 Y"格式，不留主观解释空间。
3. **覆盖正向和负向**：不仅定义"应该做什么"，也定义"不应该做什么"。
4. **与需求对齐**：每条验收条件追溯到一个需求或用户故事。
5. **门禁思维**：验收条件是发布门禁，全部通过才能发布。

## 执行流程

### 1. 从需求提取验收点
- 逐条审查需求/用户故事
- 对每条需求问："怎么算做完了？"
- 列出所有可验证的条件

### 2. 用 Given-When-Then 格式书写
```
Given（前置条件）
When（执行动作）
Then（期望结果）
```
示例：
```
Given 用户已登录且购物车有 1 件商品
When 用户点击"结算"
Then 跳转到支付页，显示商品总价（含运费）
```

### 3. 覆盖各维度
| 维度 | 验收条件示例 |
|---|---|
| **功能** | 核心操作能完成预期行为 |
| **边界** | 空输入、极值、非法格式有合理处理 |
| **错误** | 异常时显示错误提示且不崩溃 |
| **权限** | 未授权用户被拒绝并引导 |
| **性能** | 响应时间在阈值内 |
| **兼容** | 在支持的浏览器/设备上正常 |

### 4. 定义负向条件
明确不应该发生的事：
- 用户不能访问他人的数据
- 提交失败时不能丢失已填表单
- 并发提交不能产生重复记录

### 5. 设置发布门禁
将验收条件分为：
- **Block（阻塞发布）**：必须通过
- **Warning（警告）**：强烈建议通过，特殊情况可豁免
- **Info（参考）**：记录但不阻塞

### 6. 转化为测试用例
每条验收条件对应至少一个测试用例（配合 `test-writer` / `qa-test-plan` 技能）。

## 验收条件质量检查

| 好的验收条件 | 坏的验收条件 |
|---|---|
| "提交后显示成功提示，3 秒后消失" | "体验要好" |
| "空邮箱提交时输入框变红并提示" | "要校验邮箱" |
| "响应时间 P95 < 500ms" | "要快" |
| "未登录用户访问 /admin 返回 401" | "要有权限控制" |

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 读需求文档 | `read_file_lines` | 提取验收点 |
| 澄清歧义 | `ask_clarification` | 确认期望行为 |
| 产出文档 | `write_file` | 写验收清单 |
| 转化为测试 | 配合 `test-writer` | 落地为自动化测试 |

## 检查清单

- [ ] 每条条件用 Given-When-Then 或可测试格式
- [ ] 覆盖功能、边界、错误、权限维度
- [ ] 包含负向条件（不应该发生什么）
- [ ] 每条追溯到具体需求
- [ ] 发布门禁分级明确
- [ ] 每条可转化为测试用例
- [ ] 无歧义（不同人读出相同理解）

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| "体验要好" | 可量化的具体条件 |
| 只写正向条件 | 覆盖负向（不应发生） |
| 全部同等优先级 | 分 Block/Warning/Info |
| 无法转化为测试 | 每条对应一个测试用例 |
| 主观判断词（"合理""适当"） | 客观可验证的描述 |

## 输出要求

1. **验收条件清单**：每条用 Given-When-Then 格式
2. **维度覆盖**：功能/边界/错误/权限/性能
3. **负向条件**：明确不应该发生的
4. **门禁分级**：Block / Warning / Info
5. **需求追溯**：每条条件对应的需求 ID
6. **测试映射**：每条条件对应的测试用例（如已知）
