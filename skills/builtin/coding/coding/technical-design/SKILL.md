# Technical Design

## 适用场景

在编码前产出技术设计：模块方案、数据流设计、接口契约、风险分析、实现策略。是需求到实现的桥梁。

## 核心原则

1. **设计先于编码**：复杂变更先出设计文档，评审后再实现。
2. **数据流驱动**：从数据如何流动和变换的角度设计，而非从函数堆砌。
3. **接口契约明确**：模块间交互的输入输出必须显式定义。
4. **风险前置**：识别技术风险和不确定性，在设计阶段就规划缓解措施。
5. **可演进**：设计应支持渐进式实现，而非一次性大爆炸。

## 执行流程

### 1. 明确设计目标
- 一句话陈述：这个设计解决什么问题
- 列出约束：性能要求、兼容性、时间预算、技术栈限制
- `read_file_lines` / `search_code` 了解相关现有代码

### 2. 数据流设计
- **输入**：数据从哪里来、什么格式
- **处理**：经过哪些变换步骤
- **存储**：中间和最终数据存哪里
- **输出**：最终产出什么、给谁
- 画数据流图（文字或 mermaid）

### 3. 模块与接口
- 拆分为可独立实现的模块
- 每个模块定义：
  - 职责（一句话）
  - 公共接口（函数签名 / API 端点 / 事件）
  - 依赖（依赖哪些其他模块）
  - 被依赖（谁会调用它）
- 接口契约文档：参数、返回值、错误情况、副作用

### 4. 关键决策与权衡
对每个有多种方案的设计点：
| 决策点 | 方案 A | 方案 B | 选择 | 理由 |
|---|---|---|---|---|

### 5. 风险分析
| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| 第三方 API 限流 | 高 | 中 | 加重试 + 缓存 |
| 数据量大时性能 | 中 | 高 | 分批处理 + 索引 |

### 6. 实现策略
- **分阶段**：MVP → 增强 → 优化
- **依赖顺序**：哪些模块必须先实现
- **验证点**：每个阶段如何验证正确性
- **回退方案**：如果设计走不通，如何回退

## 设计文档结构

```markdown
# 技术设计：<标题>

## 1. 背景与目标
## 2. 约束与假设
## 3. 数据流
## 4. 模块设计
  ### 4.1 模块 A
  - 职责：
  - 接口：
  - 依赖：
  ### 4.2 模块 B
## 5. 关键决策（ADR）
## 6. 风险与缓解
## 7. 实现计划
## 8. 开放问题
```

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 了解现有代码 | `read_file_lines` / `search_code` | 设计基于现状 |
| 写设计文档 | `write_file` | 产出设计 |
| 评审接口 | `find_symbols` | 确认现有接口 |

## 检查清单

- [ ] 设计目标一句话清晰
- [ ] 数据流已画出
- [ ] 每个模块职责和接口已定义
- [ ] 关键决策有权衡记录
- [ ] 风险已识别并有缓解措施
- [ ] 实现可分阶段（非大爆炸）
- [ ] 开放问题已列出
- [ ] 设计与现有代码兼容

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 直接编码无设计 | 先出设计再实现 |
| 接口模糊"到时候再说" | 显式定义接口契约 |
| 只讲怎么做不讲为什么 | 记录决策理由 |
| 忽略风险 | 识别+缓解 |
| 一次性实现全部 | 分阶段可验证 |

## 输出要求

1. **设计文档**：按标准结构产出
2. **数据流图**：文字描述或 mermaid
3. **接口契约**：模块间的公共接口清单
4. **决策记录**：关键 ADR
5. **风险清单**：风险 + 缓解措施
6. **实现计划**：分阶段步骤 + 验证点
