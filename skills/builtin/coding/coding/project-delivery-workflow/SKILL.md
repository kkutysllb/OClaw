# Project Delivery Workflow

## 适用场景

- 从需求到交付的完整项目流程编排
- 多阶段、多模块的复杂项目需要系统性推进
- 需要协调设计、开发、测试、部署、文档等多个环节

## 核心原则

1. **阶段分明**：每个阶段有明确的输入、输出和验收标准
2. **前置依赖**：后一阶段依赖前一阶段的输出，不跳步
3. **渐进明细**：设计阶段不需要100%详细，但核心决策要明确
4. **持续验证**：每个阶段结束都有验证环节，不积累风险
5. **可追溯**：从需求到部署，每步决策都有记录

## 执行流程

### 阶段 1：需求分析（Requirements）

**输入**：用户需求描述
**活动**：
- 澄清模糊需求，确认边界
- 识别用户角色和使用场景
- 列出功能清单和非功能需求
- 评估技术可行性和风险
**输出**：需求文档 / 用户故事
**验收**：用户确认需求准确

### 阶段 2：技术设计（Design）

**输入**：需求文档
**活动**：
- 系统架构设计（模块划分、数据流）
- 技术栈选择
- API 设计（端点、schema）
- 数据模型设计
- 关键风险和缓解方案
**输出**：技术设计文档
**验收**：方案可覆盖所有需求

### 阶段 3：项目脚手架（Scaffold）

**输入**：技术设计文档
**活动**：
- 创建项目结构（目录、配置）
- 初始化依赖和工具链
- 搭建 CI/CD 基础
- 配置开发环境
**输出**：可运行的项目骨架
**验收**：`dev` 命令可启动，`test` 可运行

### 阶段 4：实现（Implement）

**输入**：技术设计 + 项目骨架
**活动**：
- 按模块/功能点逐步实现
- 每个功能配对应的测试
- 定期 commit，保持代码可回溯
- 持续集成验证
**输出**：功能完整的代码
**验收**：所有功能点实现，测试通过

### 阶段 5：验证（Validate）

**输入**：实现代码
**活动**：
- 单元测试覆盖率检查
- 集成测试和 E2E 测试
- 性能基准测试
- 安全审查
- 手动验收测试
**输出**：测试报告
**验收**：所有 P0/P1 测试通过

### 阶段 6：代码审查（Review）

**输入**：实现代码 + 测试报告
**活动**：
- 代码质量审查（可维护性、一致性）
- 架构符合度检查
- 安全审查
- 性能审查
**输出**：审查报告 + 修复
**验收**：审查发现已修复

### 阶段 7：文档（Document）

**输入**：实现代码 + 审查报告
**活动**：
- 更新 README
- 编写 API 文档
- 编写交接文档
- 更新架构文档
- 编写部署文档
**输出**：完整文档
**验收**：文档与代码一致

### 阶段 8：部署（Deploy）

**输入**：代码 + 文档
**活动**：
- 准备生产环境配置
- 执行部署
- 冒烟测试
- 监控确认
**输出**：线上可用的系统
**验收**：冒烟测试通过，监控正常

### 阶段 9：交接（Handoff）

**输入**：线上系统 + 文档
**活动**：
- 编写交接文档
- 培训接手者
- 知识转移
**输出**：交接完成
**验收**：接手者可独立维护

## 阶段间流转

```
需求 → 设计 → 脚手架 → 实现 → 验证 → 审查 → 文档 → 部署 → 交接
 ↑                                                        |
 └─────── 如发现问题，回退到对应阶段 ◄────────────────────┘
```

### 质量门禁

| 阶段流转 | 门禁条件 |
|---------|---------|
| 设计 → 脚手架 | 设计方案覆盖所有需求 |
| 脚手架 → 实现 | 项目可运行，CI 可用 |
| 实现 → 验证 | 所有功能点实现 |
| 验证 → 审查 | P0/P1 测试通过 |
| 审查 → 文档 | 审查发现已修复 |
| 文档 → 部署 | 文档完整且准确 |
| 部署 → 交接 | 冒烟测试通过 |

## 工具优先级

| 工具 | 用途 |
|------|------|
| 全部工具 | 根据阶段使用对应工具 |
| TodoWrite | 追踪各阶段进度 |
| `write_file` | 编写各阶段文档 |
| `git_commit` | 阶段性提交 |

## 检查清单

- [ ] 需求已确认并文档化
- [ ] 技术设计覆盖所有需求
- [ ] 项目骨架可运行
- [ ] 所有功能点已实现
- [ ] P0/P1 测试全部通过
- [ ] 代码审查发现已修复
- [ ] 文档完整且与代码一致
- [ ] 部署成功且冒烟测试通过
- [ ] 交接文档已编写

## 反模式

| ❌ 避免 | ✅ 应该 |
|---------|--------|
| 跳过设计直接编码 | 先设计再实现 |
| 不写测试 | 每个功能配测试 |
| 积累到最后才测试 | 每阶段结束就验证 |
| 文档最后才写 | 实现时同步更新 |
| 不做代码审查 | 提交前自审 + 交叉审查 |
| 一步到位不分阶段 | 按阶段推进，每步可验收 |

## 输出要求

1. 按阶段推进，每阶段有明确产出
2. 阶段间有质量门禁
3. 发现问题及时回退修正
4. 最终交付：可运行的系统 + 完整文档 + 测试通过
