# Project Delivery Workflow

> **与 Qiongqi 引擎的 7 阶段完全对齐。**
> 阶段流转由引擎的 `suggest_delivery_stage` 工具驱动，用户确认后推进。

## 适用场景

- 从需求到交付的完整项目流程编排
- 多阶段、多模块的复杂项目需要系统性推进
- 需要协调设计、开发、测试、部署、文档等多个环节

## 核心原则

1. **阶段分明**：每个阶段有明确的输入、输出和验收标准
2. **前置依赖**：后一阶段依赖前一阶段的输出，不跳步
3. **渐进明细**：设计阶段不需要100%详细，但核心决策要明确
4. **持续验证**：每个阶段结束都有验证环节，不积累风险
5. **可追溯**：从需求到交付，每步决策都有记录
6. **主动推进**：满足阶段完成信号时，主动调用 `suggest_delivery_stage`

## 七阶段定义（与引擎 delivery_stages 对齐）

### 阶段 1：需求（requirements）

**目标**：明确用户、目标、范围和验收标准。

**活动**：
- 澄清模糊需求，确认边界和非目标
- 识别用户角色和核心使用场景
- 列出功能清单和非功能需求（性能、安全、兼容性）
- 评估技术可行性和风险

**产出**：`requirements.md` / 用户故事 / 验收标准

**推荐技能**：`requirements-analysis`, `product-spec`, `acceptance-criteria`

**完成信号**：
- 需求文档已生成
- 用户角色、核心场景、验收标准均已明确写下
- 用户与 agent 已就需求范围达成一致

### 阶段 2：设计（design）

**目标**：确定架构、模块边界、数据模型和 API 合约。

**活动**：
- 系统架构设计（模块划分、依赖方向、数据流）
- 技术栈选择（框架、数据库、消息队列等）
- API 设计（端点、请求/响应 schema、错误码、版本策略）
- 数据模型设计（表结构、索引、迁移策略）
- 关键风险识别和缓解方案

**产出**：`design.md` / 架构图 / API 合约文档

**推荐技能**：`technical-design`, `architecture`, `api-design`, `database`

**完成信号**：
- 技术设计文档已生成
- 模块边界与依赖关系已明确
- 数据模型 / API 合约已定义
- 技术栈选型已与用户确认

### 阶段 3：初始化（initialization）

**目标**：建立可运行、可测试、可构建的项目骨架。

**活动**：
- 创建项目目录结构（src、tests、docs、config）
- 初始化包管理文件（pyproject.toml / package.json / go.mod / Cargo.toml）
- 搭建构建系统和测试入口
- 配置 CI 基础命令（lint + test）
- 编写 README 和开发环境说明

**产出**：可运行的项目骨架 + CI 基础配置

**推荐技能**：`project-scaffolding`, `environment-setup`, `build-system`, `ci-cd`

**完成信号**：
- 项目骨架已生成（目录结构 / 包管理文件 / 入口文件）
- 环境配置与依赖列表已写入项目
- 构建命令和测试入口可运行

### 阶段 4：实现（implementation）

**目标**：按垂直切片交付可验证功能。

**活动**：
- 按垂直切片实现功能（UI → API → service → persistence → test）
- 每个切片配对应的测试（TDD 优先）
- 定期 commit，保持代码可回溯（Conventional Commits）
- 持续运行 lint + test 确保不引入回归

**产出**：功能完整的代码 + 测试

**推荐技能**：`vertical-slice-development`, `implement`, `test-driven-development`

**完成信号**：
- 本轮规划的全部垂直切片功能均已提交
- 新增代码伴随必要的单元测试
- 用户需求中要求的核心功能可用

### 阶段 5：验证（verification）

**目标**：用自动化测试和浏览器验证确认功能可用。

**活动**：
- 运行完整测试套件，确认全部通过
- 检查测试覆盖率，补充关键路径测试
- 浏览器手动验证关键交互路径（前端项目）
- 检查错误状态、边界场景和回归风险
- 性能基准测试（如有性能要求）

**产出**：测试报告 + 验证记录

**推荐技能**：`qa-test-plan`, `webapp-testing`, `playwright-verification`

**完成信号**：
- 自动化测试全部通过
- 浏览器手动验证已记录关键路径可用
- 错误状态与边界场景已检查

### 阶段 6：审查（review）

**目标**：审查 diff、PR 风险、安全风险和可维护性。

**活动**：
- 代码质量审查（可维护性、一致性、复杂度）
- 架构符合度检查（实现是否偏离设计）
- 安全审查（认证、授权、输入校验、密钥泄露）
- 性能审查（查询效率、内存使用、渲染性能）
- 输出审查报告并修复阻塞性问题

**产出**：审查报告 + 修复

**推荐技能**：`diff-analysis`, `code-review`, `security-review`, `pr-review-advanced`

**完成信号**：
- 代码审查已完成并输出问题清单
- 阻塞问题均已修复或已有明确的跟进计划
- 安全 / 可维护性风险已评估

### 阶段 7：交付（delivery）

**目标**：准备发布、部署、运维和交接材料。

**活动**：
- 编写/更新 `DEPLOYMENT.md`（部署步骤、环境变量、配置项）
- 准备发布检查清单和回滚方案
- 编写运维 runbook（监控、告警、故障排查）
- 编写交接文档（架构说明、维护指南、已知问题）
- 更新 CHANGELOG 和版本号

**产出**：部署文档 + 运维 runbook + 交接材料

**推荐技能**：`deployment`, `release-engineering`, `operations-runbook`, `handoff-docs`

**完成信号**：
- 部署 / 发布文档已生成
- 运维 runbook 和回滚方案已写入
- 交接材料准备完毕

## 阶段间流转

```
需求 → 设计 → 初始化 → 实现 → 验证 → 审查 → 交付
 ↑                                         |
 └─────── 如发现问题，回退到对应阶段 ◄──────┘
```

**流转规则**：
- 每个阶段的完成信号满足时，主动调用 `suggest_delivery_stage`
- 用户是最终裁决者——agent 建议，用户确认
- `delivery` 是终态，不可自动进入，必须用户手动确认
- 允许回退：发现问题可退回任意前置阶段修正

### 质量门禁

| 阶段流转 | 门禁条件 |
|---------|---------|
| 需求 → 设计 | 需求文档已生成，用户已确认范围 |
| 设计 → 初始化 | 设计方案覆盖所有需求，技术栈已确认 |
| 初始化 → 实现 | 项目可运行，构建和测试命令可用 |
| 实现 → 验证 | 所有规划的功能切片已提交 |
| 验证 → 审查 | 自动化测试全部通过 |
| 审查 → 交付 | 审查发现已修复 |

## 检查清单

- [ ] 需求已确认并文档化
- [ ] 技术设计覆盖所有需求
- [ ] 项目骨架可运行
- [ ] 所有功能切片已实现并测试
- [ ] 自动化测试全部通过
- [ ] 代码审查发现已修复
- [ ] 部署 / 运维 / 交接文档完整

## 反模式

| ❌ 避免 | ✅ 应该 |
|---------|--------|
| 跳过设计直接编码 | 先设计再实现 |
| 不写测试 | 每个功能配测试 |
| 积累到最后才测试 | 每阶段结束就验证 |
| 文档最后才写 | 实现时同步更新 |
| 不做代码审查 | 提交前自审 + 交叉审查 |
| 一步到位不分阶段 | 按阶段推进，每步可验收 |
| 等用户说"推进阶段" | 满足信号就主动建议 |
