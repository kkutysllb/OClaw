# Product Spec

## 适用场景

撰写产品规格说明：功能简报、用户故事、工作流定义、MVP 范围、产品决策记录。

## 核心原则

1. **问题导向**：规格从要解决的问题出发，而非从解决方案出发。
2. **用户故事驱动**：用"作为 X，我想 Y，以便 Z"表达需求。
3. **MVP 边界清晰**：明确什么是最小可用版本，什么可以后续迭代。
4. **决策有记录**：产品决策记录为什么这么选，便于回溯。
5. **可衡量**：每条规格有成功指标，能判断是否达成。

## 执行流程

### 1. 问题陈述
- **目标用户**：谁遇到了问题
- **问题场景**：在什么情况下遇到什么困难
- **现状痛点**：现在的解决方案有什么不好
- **期望结果**：解决后用户的体验是什么样的

### 2. 用户故事
格式：`作为 <角色>，我想要 <功能>，以便 <价值>`
- 列出所有核心角色的故事
- 每个故事标注优先级：P0（必须）/ P1（重要）/ P2（可选）
- 标注 MVP 范围

### 3. 工作流定义
- **主流程**：用户完成目标的标准路径
- **状态机**：实体在各状态间的转换条件
- **触发条件**：什么动作触发什么流程

### 4. 成功指标
- **定量**：转化率、完成时间、错误率
- **定性**：用户满意度、可用性评分

### 5. 范围与优先级
| 功能 | 优先级 | MVP | 说明 |
|---|---|---|---|
| 登录 | P0 | ✅ | 基础功能 |
| 第三方登录 | P1 | ❌ | V2 |
| 记住我 | P2 | ❌ | Backlog |

### 6. 产品决策记录
记录关键产品决策及理由：
- 为什么选 A 方案不选 B
- 基于什么数据或假设
- 什么情况下会重新评估

## 产品规格文档结构

```markdown
# 产品规格：<功能名>

## 1. 问题陈述
## 2. 目标用户
## 3. 用户故事
## 4. 工作流
## 5. 成功指标
## 6. 范围与优先级（MVP）
## 7. 产品决策记录
## 8. 开放问题
```

## 工具优先级

| 场景 | 工具 | 用途 |
|---|---|---|
| 了解技术可行性 | `read_file_lines` / `search_code` | 评估实现成本 |
| 澄清需求 | `ask_clarification` | 确认产品方向 |
| 产出文档 | `write_file` | 写规格 |

## 检查清单

- [ ] 问题陈述聚焦"为什么"
- [ ] 用户故事覆盖所有核心角色
- [ ] 工作流含主流程和异常
- [ ] MVP 边界清晰
- [ ] 成功指标可衡量
- [ ] 产品决策有记录和理由
- [ ] 开放问题已列出

## 反模式

| ❌ 避免 | ✅ 应该 |
|---|---|
| 从解决方案出发 | 从问题出发 |
| 用户故事无优先级 | P0/P1/P2 标注 |
| MVP 范围模糊 | 明确"本次做什么" |
| 无成功指标 | 可衡量的指标 |
| 决策无记录 | 记录为什么这么选 |

## 输出要求

1. **问题陈述**：目标用户 + 痛点 + 期望
2. **用户故事清单**：含优先级和 MVP 标注
3. **工作流**：主流程 + 状态转换
4. **成功指标**：定量 + 定性
5. **范围表**：功能 + 优先级 + MVP
6. **决策记录**：关键决策及理由
